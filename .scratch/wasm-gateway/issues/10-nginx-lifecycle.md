# 10 — nginx Lifecycle 관리 (start/stop/status)

## What to build

`gw-admin start`가 nginx.conf를 생성하고 nginx를 child process로 spawn해서 소유한다. `gw-admin stop`이 nginx에 stop 시그널을 보내고 프로세스를 종료한다. `gw-admin status`가 nginx 프로세스 상태를 출력한다.

통합 테스트 스크립트와 justfile을 `gw-admin`을 사용하도록 업데이트한다.

## Agent Brief

**Category:** enhancement
**Summary:** `gw-admin start`가 nginx를 child process로 spawn하고 소유한다. `stop`/`status`로 lifecycle을 관리한다

**Current behavior:**
nginx는 `just nginx` 커맨드로 직접 실행된다. 통합 테스트 스크립트도 nginx를 직접 기동한다. Admin Process와 nginx 사이에 lifecycle 연결이 없다.

**Desired behavior:**
`gw-admin start`가 nginx.conf를 생성한 뒤 nginx를 child process로 spawn한다. 데몬으로 동작하며 nginx 프로세스를 소유한다. `gw-admin stop`이 nginx에 stop 시그널을 전송한다. `gw-admin status`가 nginx 실행 여부와 PID를 출력한다. 통합 테스트와 justfile이 `gw-admin`을 통해 nginx를 기동한다.

nginx 실행 시 working directory는 `gateway/nginx/`여야 한다 (wasm 모듈 경로가 상대 경로로 참조되기 때문).

**Key interfaces:**
- `gw-admin start [--config-dir <path>] [--nginx-prefix <path>]` — config 로드 → nginx.conf 생성 → nginx spawn
- `gw-admin stop` — nginx에 `nginx -s stop` 전송
- `gw-admin status` — nginx PID 파일 또는 프로세스 확인 후 상태 출력
- justfile: `nginx` 레시피 → `gw-admin start`, `stop` 레시피 → `gw-admin stop`
- 통합 테스트 스크립트: nginx 직접 기동 코드를 `gw-admin start` 호출로 교체

**Acceptance criteria:**
- [ ] `gw-admin start` 실행 후 `curl localhost:9000/api/v1/`이 응답한다
- [ ] `gw-admin stop` 실행 후 nginx 프로세스가 종료된다
- [ ] `gw-admin status`가 실행 중/중지됨 상태를 출력한다
- [ ] `just integration`이 `gw-admin`을 통해 nginx를 기동하고 12/12 통과한다
- [ ] `just nginx`, `just stop` 커맨드가 `gw-admin`을 래핑한다

**Out of scope:**
- 파일 감시 / 자동 reload (이슈 #11)
- Admin REST API
